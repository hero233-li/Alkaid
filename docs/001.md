我把这条链路完整顺下来了。先说结论：

> `tasks.py` 本身只负责“启动任务、校验、调用执行器、更新任务状态”。
> **决定执行哪个产品文件/处理器的，是 Job 里保存的 `execution_config_snapshot.handler`；真正控制请求先后顺序的，是 `BaseProductApplicationHandler.execute()`。**

而且这里有一个很关键的细节：

> 配置里的 `operation`，例如 `mock_product.product_a.dynamic_apply`，目前只是配置标识和结果信息，并没有通过它动态调用某个 Python 文件或函数。

# 一、整体调用链

当前完整流程是：

```text
前端提交产品申请
        ↓
views.py
create_product_application()
        ↓
根据 product + applicationMethod
生成 execution_snapshot
        ↓
保存 Job
        ↓
Celery 调用
execute_product_application.delay(job.id)
        ↓
tasks.py
execute_product_application()
        ↓
ProductApplicationExecutor.execute()
        ↓
读取 snapshot.handler
        ↓
get_product_handler(handler)
        ↓
找到具体 Handler 类
        ↓
BaseProductApplicationHandler.execute()
        ↓
按照固定顺序调用接口
```

当前真正的请求顺序是：

```text
登录
 ↓
产品检查
 ↓
更新 Token
 ↓
提交产品申请
 ↓
审计
```

也就是：

```python
self.login(context)

self.check_product(context)

self.rotate_token(context)

application = self.submit_application(context)

self.audit(context)
```

这个顺序就在：

```text
Alkaid-python/
└── apps/
    └── product_data/
        └── handlers/
            └── base.py
```

里面。

# 八、任务执行时如何决定使用哪个 Handler

`tasks.py` 调用：

```python
execution_snapshot = (
    ProductApplicationExecutor.resolve_snapshot(
        job,
        job.product,
    )
)
```

进入：

```python
@staticmethod
def resolve_snapshot(
    job,
    product_code,
):
    if job.execution_config_snapshot:
        return (
            ProductExecutionSnapshot
            .model_validate(
                job.execution_config_snapshot
            )
        )

    return (
        load_execution_catalog()
        .snapshot(product_code)
    )
```

这里的逻辑非常重要。

优先使用：

```python
job.execution_config_snapshot
```

而不是重新读取最新配置。

也就是说：

假设上午创建产品 A 任务，当时：

```json
{
  "handler": "whitelist_application_v1"
}
```

下午你修改配置：

```json
{
  "handler": "whitelist_application_v2"
}
```

上午已经创建的任务仍然执行：

```text
whitelist_application_v1
```

不会因为配置修改而改变。

这是为了避免：

```text
创建任务时是一套配置

↓

执行任务时配置突然变了

↓

重试时又使用另一套逻辑
```

Job 会保存当时的配置版本和完整快照。

# 十三、Handler 注册表决定执行哪个类

进入：

```text
apps/product_data/
└── handlers/
    └── registry.py
```

这里建立了映射表：

```python
HANDLERS = {
    "whitelist_application_v1":
        WhitelistApplicationHandler,

    "red_shield_application_v1":
        RedShieldApplicationHandler,

    "credit_application_v1":
        CreditApplicationHandler,
}
```

实际代码是自动读取类的 `code`：

```python
HANDLERS = {
    handler.code: handler
    for handler in (
        WhitelistApplicationHandler,
        RedShieldApplicationHandler,
        CreditApplicationHandler,
    )
}
```

然后：

```python
def get_product_handler(
    code: str,
):
    handler = HANDLERS[code]

    return handler()
```

所以：

```python
get_product_handler(
    "whitelist_application_v1"
)
```

返回：

```python
WhitelistApplicationHandler()
```

# 十五、真正决定请求顺序的位置

三个产品 Handler 都继承：

```python
BaseProductApplicationHandler
```

它们自己没有重写：

```python
execute()
```

因此最终执行父类：

```python
BaseProductApplicationHandler.execute()
```

顺序固定是：

```python
self.login(context)
```

↓

```python
self.check_product(context)
```

↓

```python
self.rotate_token(context)
```

↓

```python
application = (
    self.submit_application(context)
)
```

↓

```python
self.audit(context)
```

画成流程：

```text
┌──────────────────┐
│ login            │
│ 获取 Flow Token   │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ check_product    │
│ 产品准入检查      │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ rotate_token     │
│ 刷新 Token        │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ submit           │
│ 正式提交产品申请  │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ audit            │
│ 调用审计系统      │
└──────────────────┘
```

这些是普通同步方法。

因此必须等：

```text
login 执行成功
```

才会执行：

```text
check_product
```

不是同时执行，也不是异步并发。

# 十七、每个请求最后如何真正发送

比如：

```python
self.login(context)
```

内部调用：

```python
ctx.call(
    "auth.login",
    LOGIN,
    {...},

```

然后：

```python
ctx.call()
```

进入：

```python
EndpointExecutor.execute()
```

同时传入：

```python
JobHttpCallObserver(
    self.job,
    step=step,

```

所以每次 HTTP 调用都会记录：

```text
请求地址

请求报文

响应报文

请求耗时

成功或失败

```

最终：

```python
EndpointExecutor.execute()
```

会：

```text
1. 根据接口配置添加 Token

2. 调用 HttpClient

3. 校验响应业务码

4. 根据接口配置更新 Token

5. 返回响应对象
```

# 十八、关于 `applicationMethod`，这里容易误解

产品 A 配置：

```json
"application_methods": {
  "normal": {
    "operation":
      "mock_product.product_a.apply"
  },

  "dynamic": {
    "operation":
      "mock_product.product_a.dynamic_apply"
  },

  "extra": {
    "operation":
      "mock_product.product_a.extra_apply"
  }
}
```

看起来像：

```text
normal
↓
调用 product_a.apply()

dynamic
↓
调用 product_a.dynamic_apply()

extra
↓
调用 product_a.extra_apply()
```

但当前实际上不是。

当前代码只是把：

```python
operation
```

保存进快照：

```python
operation=method.operation
```

然后执行完成时放进返回结果：

```python
"operation": snapshot.operation
```

没有类似：

```python
import_module(
    snapshot.operation
)
```

也没有：

```python
OPERATIONS[
    snapshot.operation
]()
```

因此当前：

```text
normal

dynamic

extra
```

主要决定：

```text
启用哪些字段

哪些字段必填

Job 中记录哪个申请方式

结果中显示哪个 operation
```

但是不会改变真正调用的 Python 方法。

目前三种申请方式最后都进入：

```python
WhitelistApplicationHandler
```

然后执行相同的：

```text
login

check_product

rotate_token

submit_application

audit
```
